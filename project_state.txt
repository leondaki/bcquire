CORE DESIGN PATTERN: strict separation of rules engine / theme / UI / (future)
networking. This is the single load-bearing decision in the codebase:
  - core/        Pure rules engine. RefCounted classes only. No Node, no UI,
                 no theme, no networking. Fully deterministic. Drives itself
                 via explicit method calls (do_found, do_grow, merger_begin,
                 buy_stock, etc.) — these are exactly the calls net/session.gd
                 now drives on the host's behalf (M4 Stage A, see section 3).
  - net/         M4 multiplayer plumbing (Stages A+B done, see section 3):
                 action.gd/event.gd (Dictionary schemas), network_transport.gd
                 (abstract interface, RefCounted), loopback_transport.gd +
                 loopback_hub.gd (zero-socket, headlessly-testable transport),
                 enet_transport.gd (real ENetMultiplayerPeer LAN transport —
                 extends Node, not NetworkTransport, since Godot's high-level
                 RPC system requires a Node; duck-typed to the same surface),
                 session.gd (GameSession — host-authoritative router between
                 GameState and whichever transport is active). core/ still
                 never imports net/ — the dependency runs net/ -> core/, one
                 direction, same as ui/ -> core/.
  - theme/       ThemeDef resource (theme_def.gd) + default_theme.tres. ALL
                 cosmetic config: board size/tile size, palette, per-chain
                 display name + color (indexed by AcqEnums.ChainId). UI reads
                 ONLY from ThemeDef; core/ never imports theme/. Re-skinning =
                 new .tres + art, zero logic changes.
  - ui/game/     The interactive game (current main scene). Controller
                 (game.gd) owns a GameSession (net/session.gd) and reads its
                 GameState mirror; drives the turn through phases by sending
                 Actions and reacting to session.event_applied — builds the
                 entire UI procedurally in code (no scene sub-nodes beyond the
                 bare Control root) for steppers, popups, cards, etc. Defaults
                 to a solo hotseat_mode session, so today's local play is
                 unchanged even though every mutation now goes through net/.
  - ui/board/    LEGACY M1 sandbox (Tile.tscn/board.gd). Superseded by
                 ui/game/. Still present on disk but not the run scene and
                 not maintained — safe to delete later, harmless to ignore now.
  - sim/         Headless test suites (SceneTree scripts, no GUT dependency).
                 Includes run_network_tests.gd (M4 Stage A): host/client
                 GameState convergence test over a loopback transport.

NEXT MILESTONES (in order):
  1. M4 — Networked multiplayer. Staged per net/session.gd's host-
     authoritative design (full plan: a now-historical plan doc; this section
     is the live status).
       STAGE A — DONE: Action/Event plumbing + loopback transport + the
         ui/game/game.gd refactor.
           - net/action.gd, net/event.gd: Action/Event are plain Dictionaries
             ({type, player, payload}); these two files hold the type
             constants and make_*() builders.
           - net/network_transport.gd: abstract interface (host/join/
             send_action/broadcast_event/send_event_to, signals
             action_received/event_received/peer_joined/peer_left).
           - net/loopback_transport.gd + net/loopback_hub.gd: in-process,
             zero-socket transport — peer id is registration order on a
             shared LoopbackHub (host is always peer 0). This is what makes
             the whole layer testable headlessly with no real networking.
           - net/session.gd (GameSession): the host-authoritative router.
             Host always independently re-runs the same GameState calls
             ui/game/game.gd used to call directly (classify_placement,
             do_found, merger_begin, etc.) and broadcasts an Event built from
             the *actual* result; clients never re-derive an outcome — they
             replay the identical state.* calls from the event's payload
             (_apply_event_to_state). This is what guarantees host/client
             convergence by construction.
           - ui/game/game.gd: every former direct state.do_xxx() call site
             (drop_tile, _on_found_chosen, _on_survivor_chosen,
             _on_dispose_confirm, _commit_buy, _redraw_tile) now builds an
             Action and calls session.send_action(); a single dispatcher
             _on_event_applied() (driven by session.event_applied) does all
             the UI-state updates that used to live inline in those
             functions. _is_my_turn()/_is_my_seat() seat-gating helpers exist
             and disable input + show a "Waiting for ...'s move" label for
             non-active seats, but are inert by default: _ready() wires up a
             solo GameSession with hotseat_mode = true (one peer, every seat
             playable locally), which is what reproduces today's hotseat
             behaviour unchanged. _generate_midgame() dev harness still pokes
             GameState directly (session.state = state) — intentionally not
             routed through Actions, since it's test tooling, not real play.
           - sim/run_network_tests.gd: headless convergence test — two
             GameSessions (host seat 0 / client seat 1) share a LoopbackHub
             and auto-play a full game purely through session.send_action(),
             asserting host/client GameState deep-equality after every single
             action. This is the regression net for the Action/Event layer;
             run it alongside the other two suites (section 4).
       STAGE B — DONE: net/enet_transport.gd + a minimal lobby row in
         game.gd's footer.
           - net/enet_transport.gd extends Node (NOT NetworkTransport's
             RefCounted base — Godot's high-level RPC system requires a Node
             in the scene tree so a peer's RPC lands on the matching node at
             the same NodePath on every machine). GDScript has no enforced
             interfaces, so it just exposes the identical method/signal
             surface NetworkTransport documents; net/session.gd's `transport`
             field is deliberately untyped (duck-typed) to accept either.
             Host/server is always peer id 1 (Godot's high-level multiplayer
             convention); client ids are ENet-assigned, NOT sequential by
             join order, so (unlike LoopbackTransport) peer id can't double
             as a seat index here.
           - Seat assignment: Event.make_game_started() gained a third
             `peer_seats` field (transport peer id -> seat index), built by
             the host's lobby (seat 0 = host, joiners get 1, 2, ... in
             connect order) and broadcast as part of the one GAME_STARTED
             event. Each client's GameSession looks itself up in that map
             (_on_event_received_as_client) and sets local_player_index —
             no separate private "you are seat N" message needed.
           - game.gd's footer gained a lobby row: host port field + Host
             button, join address/port fields + Join button, a status label,
             and a host-only "Start Networked Game" button. Host/Join replace
             the default solo `session` with a real EnetTransport-backed one
             (hotseat_mode = false) and null out `state` until the real
             GameState arrives via GAME_STARTED (can_drag_tiles/
             can_drop_tile/_refresh_all all guard against state == null —
             the board just freezes rather than crashing while a peer waits
             in the lobby). The dev-tooling row (Generate Mid-Game/Reset)
             auto-hides once any remote peer has joined
             (_dev_tools_visible()), since those buttons poke GameState
             directly and would desync anyone else watching.
           - No new headless test for Stage B by design: real socket/RPC
             behavior can't be exercised without an actual transport, so
             verification here is the manual two-window LAN checklist
             (now-historical plan doc has the original script; section 4
             below keeps the live version).

4. CRITICAL CONTEXT
================================================================================
RULES CORRECTNESS (load-bearing, do not regress):
  - Merger survivor/bonus sizes are computed from PRE-merger chain sizes —
    the tile that causes the merger is NEVER counted toward either chain's
    size when picking the survivor or computing bonuses. Enforced by
    test_merger_excludes_new_tile() in sim/run_tests.gd. This is the single
    easiest Acquire rule to get wrong; do not "simplify" merger code without
    re-running that test.
  - "Closest to 1A" for starting-tile turn order is ROW-MAJOR, not column-
    major: rank = row * BOARD_WIDTH + col. This was gotten wrong on the
    first implementation attempt (used col-major) and caught only because
    the rules text's own example ("9A is closer to 1A than 1B") was encoded
    as an explicit test assertion. If this logic is ever touched again,
    re-derive from that exact sentence in ACQUIRE RULES.txt, don't trust
    intuition.
  - Bonus rounding: ties round UP to the nearest $100 (StockMarket-adjacent
    helper _round_up_100() in game_state.gd). This is in the official rules
    text verbatim ("rounded up to the nearest hundred").
  - The price/bonus formulas were independently validated against the rules
    doc's own Tycoon-mode worked example (Imperial, size 5: price 700,
    primary/majority 7000, tertiary 3500) even though this project only
    implements Classic Mode (2 payout tiers: majority/minority, no
    tertiary) — Classic Mode's "minority" bonus is mathematically the same
    5x-price tier as Tycoon Mode's "tertiary" in that example, which is how
    the cross-check works. Tycoon Mode (3-tier bonus) is NOT implemented and
    not currently planned.

DELIBERATE RULE DEVIATION (user-requested, keep as is):
  - Strict rules say unplayable ("dead") tiles are only set aside at END of
    turn, and any newly-dead tile from that turn's draw must wait until next
    turn to be discarded. This project instead gives the player an immediate
    "Redraw" button next to any dead tile in their rack, usable any time
    during PLACE_TILE phase. This was an explicit user request (QoL during
    testing), not an oversight — do not "fix" this to match strict rules
    without asking first.

ENGINE GOTCHAS HIT AND FIXED (so they aren't rediscovered):
  - GDScript type inference: untyped Array literals used in typed contexts
    (e.g. `const DIRS := [Vector2i(...), ...]`) fail to compile under strict
    typing with "Cannot infer the type of variable" — must declare as
    `Array[Vector2i]` explicitly. Hit once in game_state.gd's DIRS constant;
    if a similarly "obviously typed" const array gets added, declare its
    element type explicitly up front.
  - BoardCell.setup() must NOT call refresh() during construction — cells
    are built (in game.gd's _build_layout) BEFORE a GameState exists; calling
    state.cell() at that point hits a null `_ctrl.state`. The controller's
    _refresh_all() (called once after setup_new_game()/_generate_midgame())
    is solely responsible for the first paint.
  - Headless SceneTree test scripts that instance a scene and immediately
    call a method on it must defer to at least one _process() tick before
    interacting with it if the scene's _ready() builds UI children — calling
    methods from _initialize() (same frame as add_child) runs before _ready,
    so children referenced by the controller are still null. See
    sim/check_midgame.gd's file-header comment.
  - Button.text does not support `&` mnemonics being escaped — a literal "&&"
    in a button label renders as a doubled "&&" on screen, not "&". Use a
    single "&" if you want a literal ampersand in a Godot Button's text.
